# Development Policy

## Цель Текущего Этапа

На текущем этапе проект должен оставаться маленьким учебным reference, который прозрачно показывает:

- установку MCP-соединения;
- вызов `initialize`;
- получение `tools/list`;
- вывод списка доступных инструментов.

## Технологический Выбор

- основной стек проекта на текущем этапе: Kotlin;
- по умолчанию не распылять реализацию между несколькими языками, если Kotlin покрывает задачу;
- при выборе библиотек и структуры проекта предпочитать решения, которые естественно живут в JVM-мире и не мешают будущей интеграции.

## Предпочтения По Архитектуре

- по умолчанию не завязывать клиента на локальный subprocess, если задача допускает HTTP-based transport;
- endpoint сервера выносить в конфиг или в явно выделенную переменную, чтобы локальный `http://localhost:...` можно было позже заменить на удалённый URL;
- transport-логику, инициализацию и демонстрационный вызов `tools/list` не смешивать в одном большом неструктурированном блоке, если это мешает расширению;
- если появляется проектно-важный orchestration-код, по умолчанию документировать его.

## Минимум Без Лишней Сложности

- не добавлять в стартовый reference лишние абстракции, которые не помогают понять базовый MCP lifecycle;
- новые расширения добавлять так, чтобы минимальный happy-path `connect -> initialize -> tools/list` оставался очевидным и легко проверяемым;
- если есть выбор между более быстрым и более сложным стартом, на текущем этапе предпочитать решение, которое быстрее даёт рабочий end-to-end результат без потери расширяемости.

## Кодировка И Консольный Вывод

- для русскоязычного клиентского вывода в Windows/PowerShell считать кодировку частью рабочего сценария, а не косметикой;
- если клиентский вывод или scripted e2e-проверка показывают кракозябры, проверять оба слоя:
  - кодировку самого приложения;
  - кодировку процесса и shell-окружения, которое захватывает вывод;
- в клиентской точке входа по умолчанию держать `stdout` и `stderr` настроенными на UTF-8 перед печатью итогового вывода;
- в PowerShell-скриптах, которые запускают клиент и читают его вывод, по умолчанию выставлять:
  - `[Console]::InputEncoding = [System.Text.Encoding]::UTF8`
  - `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8`
  - `$OutputEncoding = [System.Text.Encoding]::UTF8`
- если клиент запускается через `cmd.exe`, перед запуском использовать `chcp 65001`, а для redirected stdout/stderr по возможности явно задавать `StandardOutputEncoding` и `StandardErrorEncoding` как UTF-8;
- признаком корректного решения считать не только зелёный exit code, но и читаемый русскоязычный вывод в ручном запуске и в scripted e2e-проверке.

## Команда Пользователя `собери проект`

- в этом проекте фраза пользователя `собери проект` по умолчанию означает не только сборку, а полный workflow ручной проверки;
- такой запрос нужно трактовать как запуск:
  - `powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1`
- то есть `собери проект` в этом репозитории означает:
  - собрать проект;
  - поднять сервер;
  - дождаться готовности endpoint;
  - открыть клиента для ручной проверки;
- если пользователю нужен именно голый `.\gradlew.bat build` без запуска сервера и клиента, он должен явно попросить только сборку.

## Команда Пользователя `запусти проект`

- в этом проекте фраза пользователя `запусти проект` по умолчанию означает тот же workflow ручной проверки, но без сборки;
- такой запрос нужно трактовать как запуск:
  - `powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1 -SkipBuild`
- то есть `запусти проект` в этом репозитории означает:
  - использовать уже собранные артефакты;
  - поднять сервер;
  - дождаться готовности endpoint;
  - открыть клиента для ручной проверки;
- если есть основания считать артефакты устаревшими или отсутствующими, лучше трактовать такой запрос как `собери проект`.
