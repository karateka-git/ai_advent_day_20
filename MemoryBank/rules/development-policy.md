# Development Policy

## Цель Текущего Этапа

На текущем этапе проект должен оставаться маленьким учебным reference, который прозрачно показывает:

- установку MCP-соединения;
- вызов `initialize`;
- получение `tools/list`;
- вывод списка доступных инструментов.

## Технологический Выбор

- основной стек проекта на текущем этапе: Kotlin;
- по умолчанию не распылять реализацию между несколькими языками, если Kotlin покрывает задачу;
- при выборе библиотек и структуры проекта предпочитать решения, которые естественно живут в JVM-мире и не мешают будущей интеграции.

## Предпочтения По Архитектуре

- по умолчанию не завязывать клиента на локальный subprocess, если задача допускает HTTP-based transport;
- endpoint сервера выносить в конфиг или в явно выделенную переменную, чтобы локальный `http://localhost:...` можно было позже заменить на удалённый URL;
- transport-логику, инициализацию и демонстрационный вызов `tools/list` не смешивать в одном большом неструктурированном блоке, если это мешает расширению;
- если появляется проектно-важный orchestration-код, по умолчанию документировать его.

## Минимум Без Лишней Сложности

- не добавлять в стартовый reference лишние абстракции, которые не помогают понять базовый MCP lifecycle;
- новые расширения добавлять так, чтобы минимальный happy-path `connect -> initialize -> tools/list` оставался очевидным и легко проверяемым;
- если есть выбор между более быстрым и более сложным стартом, на текущем этапе предпочитать решение, которое быстрее даёт рабочий end-to-end результат без потери расширяемости.
