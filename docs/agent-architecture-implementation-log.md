Принятые решения:

- сначала создать устойчивые границы слоёв и только после этого переносить живую логику.

## Этап 2. Выделение MCP-Слоя

Статус: завершён

Цель этапа:

- изолировать работу с Kotlin MCP SDK внутри `mcp` слоя;
- перевести текущий сценарий `connect` на проектный `McpClient`;
- сделать проектные MCP-модели основным контрактом для верхних слоёв.

Выполненные действия:

1. В архитектурном ТЗ детализированы подэтапы `Этапа 2`, чтобы зафиксировать целевой vertical slice и критерии переноса клиентской логики.
2. Добавлена реализация `DefaultMcpClient`, которая:
   - создаёт HTTP client с SSE;
   - поднимает Streamable HTTP transport;
   - создаёт SDK client;
   - выполняет подключение и `tools/list`;
   - возвращает проектный `McpConnectionSnapshot`.
3. Локализованы SDK-зависимые преобразования в `mcp` слое:
   - server info -> `McpServerInfo`;
   - tool -> `McpToolDescriptor`.
4. Текущий клиентский сценарий `connect` переведён на использование `McpClient` вместо прямой работы с Kotlin MCP SDK.
5. Локальные unit-проверки клиента обновлены так, чтобы проверять форматирование через `McpToolDescriptor`, а не через SDK-типы.

Принятые решения:

- на `Этапе 2` не протягивать ещё `application` и `agent` в runtime-flow, а ограничиться чистым переносом интеграции с MCP в отдельный слой;
- держать `DefaultMcpClient` самодостаточным и не выносить пока дополнительные фабрики transport/http client, чтобы не раздувать абстракции раньше времени;
- сохранить совместимое внешнее поведение команды `connect`, чтобы рефакторинг оставался поэтапным и безопасным.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1` завершается успешно;
- e2e-проверка подтверждает, что `runClient` по-прежнему выводит server info и список инструментов `ping`/`echo`.

Коммиты этапа:

- `4965769` — детализация шагов `Этапа 2` в архитектурном ТЗ;
- `a79d3bf` — добавление `DefaultMcpClient` как реализации проектного MCP-слоя;
- `93d3e1d` — перевод текущего сценария `connect` на `McpClient` и обновление тестов.

Следующий шаг:

- перейти к `Этапу 3` и поставить полноценный агентный слой между `application` и `mcp`, чтобы orchestration `connect` начал жить уже в `agent`, а не в клиентском entrypoint.

## Этап 3. Внедрение Агентного Слоя

Статус: завершён

Цель этапа:

- поставить рабочий агентный слой между верхним runtime-flow и `mcp`;
- перенести orchestration `connect` в `agent`;
- подтвердить, что агент использует только проектный `McpClient`, а не SDK напрямую.

Выполненные действия:

1. В архитектурном ТЗ детализированы подэтапы `Этапа 3`, чтобы зафиксировать границу ответственности агента и ожидаемый vertical slice.
2. Добавлена рабочая реализация `DefaultAgent`, которая:
   - принимает `AgentRequest`;
   - для `AgentRequest.Connect` вызывает `McpClient`;
   - возвращает `AgentResponse.ConnectSuccess` или `AgentResponse.Failure`.
3. Текущий клиентский сценарий `connect` переведён на использование `Agent` вместо прямого вызова `McpClient`.
4. Добавлены unit-проверки на агентный orchestration-слой:
   - успешный `connect`;
   - ошибка при исключении в `McpClient`.
5. Тесты стабилизированы без введения новых зависимостей на coroutine-test: использован `runBlocking`.

Принятые решения:

- на `Этапе 3` не протягивать ещё `application` в runtime-flow, а сначала сделать реальным именно агентный слой;
- оставить `AgentResponse` простым и сфокусированным на `connect`, но уже в форме, которую позже можно расширять под resources и tool calls;
- на ошибке агента пока завершать клиентский сценарий исключением, чтобы не менять прежний внешний смысл `runClient` до внедрения `application/result` на следующем этапе.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1` завершается успешно;
- e2e-проверка подтверждает, что после встраивания агента пользовательский сценарий `connect` по-прежнему даёт server info и список `ping`/`echo`.

Коммиты этапа:

## Этап 4. Внедрение Application-Слоя Команд

Статус: завершён

Цель этапа:

- встроить рабочий `application` слой поверх агента;
- перевести runtime-flow на `Command -> ApplicationCommandHandler -> Agent -> McpClient`;
- сделать `CommandResult` основным контрактом верхнего слоя вместо `AgentResponse`.

Выполненные действия:

1. В архитектурном ТЗ детализированы подэтапы `Этапа 4`, чтобы зафиксировать vertical slice через `Command` и `CommandResult`.
2. Добавлена рабочая реализация `DefaultApplicationCommandHandler`, которая:
   - принимает `ConnectCommand`;
   - вызывает `Agent`;
   - преобразует `AgentResponse.ConnectSuccess` в успешный `ConnectResult`;
   - преобразует `AgentResponse.Failure` в неуспешный `ConnectResult`.
3. Добавлены unit-проверки application-слоя:
   - успешное выполнение `ConnectCommand`;
   - перевод агентной ошибки в `ConnectResult` с `connected = false`.
4. Текущий клиентский сценарий `connect` переведён на использование `ApplicationCommandHandler` вместо прямого вызова `Agent`.

Принятые решения:

- не расширять пока `Command` и `CommandResult` дополнительными ветками, а отработать до конца один полноценный `connect` вертикальный срез;
- сохранить прежний пользовательский смысл `runClient`: при неуспешном `ConnectResult` entrypoint всё ещё завершает сценарий исключением;
- до выделения presentation-слоя использовать в `ConnectCommand` явный `endpointOverride`, чтобы application-слой уже был реален, но не зависел от CLI-аргументов напрямую.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1` завершается успешно;
- e2e-проверка подтверждает, что после встраивания `application` слоя `runClient` по-прежнему выводит server info и список `ping`/`echo`.

Коммиты этапа:

- `5a91cef` — детализация шагов `Этапа 4` в архитектурном ТЗ;
- `a195a08` — добавление `DefaultApplicationCommandHandler` и unit-проверок application-слоя;
- `42125d9` — перевод текущего `connect` flow на application-слой.

Следующий шаг:

## Этап 5. Выделение CLI Presentation-Слоя

Статус: завершён

Цель этапа:

- сделать CLI тонким presentation-адаптером поверх новой архитектуры;
- перевести входной разбор в `CliCommandParser`;
- перевести текстовый вывод в `CliOutputFormatter`;
- упростить `main` до composition root.

Выполненные действия:

1. В архитектурном ТЗ детализированы подэтапы `Этапа 5`, чтобы зафиксировать границу presentation-слоя и итоговый vertical slice.
2. Добавлена рабочая реализация `DefaultCliCommandParser`, которая:
   - понимает команду `connect`;
   - подставляет endpoint по умолчанию;
   - возвращает `ConnectCommand`.
3. Добавлена рабочая реализация `DefaultCliOutputFormatter`, которая:
   - форматирует успешный `ConnectResult`;
   - форматирует неуспешный `ConnectResult`;
   - не зависит от `AgentResponse` и MCP SDK.
4. `McpClientApp.kt` переписан в форме composition root:
   - настройка UTF-8;
   - создание parser/handler/formatter;
   - последовательный вызов `parse -> handle -> format`.
5. Локальные тесты CLI вынесены на уровень parser и formatter:
   - разбор `connect`;
   - ошибка на неизвестной команде;
   - форматирование успешного и неуспешного `ConnectResult`.

Принятые решения:

- на этом этапе не вводить ещё отдельный UI-specific error channel, а оставить formatter единой точкой превращения `CommandResult` в консольный текст;
- подстановку endpoint по умолчанию сделать зависимостью parser через функцию, чтобы presentation-слой не был жёстко завязан на один способ конфигурации;
- старые helper-функции форматирования из `McpClientApp.kt` убрать, чтобы не дублировать presentation-логику в entrypoint.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1` завершается успешно;
- e2e-проверка подтверждает, что после выделения parser/formatter пользовательский вывод `runClient` остаётся совместимым.

Коммиты этапа:
