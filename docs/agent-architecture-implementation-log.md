## Этап 1. Архитектурный Каркас Слоёв

Статус: завершён

Цель этапа:

- подготовить документационную и кодовую рамку для нового рефакторинга;
- завести пакеты и базовые контракты слоёв `application`, `agent`, `mcp`, `presentation`;
- подтвердить, что после добавления архитектурного каркаса проект остаётся рабочим.

Выполненные действия:

1. Под архитектурный рефакторинг создано отдельное ТЗ `agent-architecture-spec.md`.
2. Старый журнал реализации переименован в `mcp-reference-implementation-log.md`, чтобы он явно относился к предыдущему MCP reference-ТЗ.
3. Создан отдельный журнал `agent-architecture-implementation-log.md` для нового ТЗ.
4. README обновлён ссылками на оба ТЗ и оба журнала реализации.
5. В архитектурном ТЗ детализированы подэтапы внедрения для `Этапа 1`.
6. В кодовой базе созданы новые пакеты и базовые контракты:
   - `application/command`
   - `application/result`
   - `application/service`
   - `agent`
   - `mcp`
   - `mcp/model`
   - `presentation/cli`
7. Добавлен минимальный набор типов каркаса:
   - `Command`
   - `ConnectCommand`
   - `CommandResult`
   - `ConnectResult`
   - `ApplicationCommandHandler`
   - `Agent`
   - `AgentRequest`
   - `AgentResponse`
   - `McpClient`
   - проектные MCP-модели
   - CLI-контракты parser/formatter
8. Добавлены unit-проверки на базовые контракты новой архитектуры.

Принятые решения:

- не смешивать новый архитектурный рефакторинг с уже завершённым reference-ТЗ по MCP sandbox;
- ввести новый каркас поверх работающего проекта, не ломая текущий runtime-flow на первом этапе;
- сначала создать устойчивые границы слоёв и только после этого переносить живую логику.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1` завершается успешно;
- текущий пользовательский сценарий `initialize -> tools/list` сохраняет рабочее поведение, несмотря на появление нового архитектурного каркаса.

Коммиты этапа:
