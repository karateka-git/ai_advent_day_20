# Техническое задание

## Название

Orchestration MCP: длинный flow с несколькими MCP-серверами и инструментами.

## Цель

Развить текущий sandbox-проект до сценария, в котором агент:

1. автоматически обнаруживает возможности нескольких MCP-серверов;
2. выбирает нужный инструмент на основе доступных capabilities;
3. корректно маршрутизирует вызовы между разными серверами;
4. выполняет длинный пользовательский flow из нескольких шагов;
5. возвращает итоговый результат как единый прикладной сценарий.

Минимально успешный результат:

- в проекте зарегистрировано несколько MCP-серверов;
- один пользовательский сценарий использует инструменты с разных серверов;
- порядок вызовов и передача данных между шагами подтверждены тестами и e2e-проверкой;
- README и проектная документация описывают новый сценарий.

## Контекст

Проект уже умеет:

- поднимать `stateless` и `stateful` MCP-серверы;
- автоматически готовить агент через capability bootstrap;
- строить список доступных пользовательских команд на основе найденных tools;
- выполнять orchestration внутри agent-layer;
- запускать scripted e2e-проверку через PowerShell.

Текущее ограничение:

- длинный flow уже существует, но фактически сосредоточен внутри одного `stateful` сервера;
- это хорошо демонстрирует композицию MCP-tools, но недостаточно явно показывает orchestration между несколькими MCP-серверами;
- routing пользовательских сценариев в основном опирается на `FixedServer`, а не на cross-server capability flow.

## Целевая модель

После доработки проект должен демонстрировать именно multi-server orchestration.

Целевой пользовательский сценарий:

- `tool summary posts <count> [strategy]`

Целевой flow:

1. агент находит инструмент получения публикаций на `stateless` сервере;
2. агент получает исходные данные с `stateless` сервера;
3. агент по внутренней логике выбирает 3 публикации по стратегии `long` или `short`;
4. агент находит инструмент объединения публикаций на `stateful` сервере;
5. агент передаёт выбранные данные на `stateful` сервер;
6. агент вызывает инструмент сохранения summary на `stateful` сервере;
7. агент возвращает единый итог выполнения pipeline;
8. отдельные команды чтения подтверждают, что результат действительно сохранён.

Ключевой смысл сценария:

- источник данных и сервер обработки/сохранения разделены;
- агент сам определяет, какой tool и на каком сервере использовать на каждом шаге;
- пользователь видит один прикладной сценарий, а не ручное управление серверами.

## Пользовательский сценарий

Основная команда:

- `tool summary posts <count> [strategy]`

Команды чтения результата:

- `tool summaries`
- `tool summary saved <summaryId>`

Примеры:

- `tool summary posts 10`
- `tool summary posts 10 long`
- `tool summary posts 10 short`
- `tool summaries`
- `tool summary saved summary-1`

Ожидаемое поведение:

- если стратегия не указана, используется `long`;
- агент использует данные минимум с двух MCP-серверов в рамках одного flow;
- итоговый вывод содержит стратегию, выбранные post ids, заголовок summary, short id и время сохранения;
- чтение из хранилища работает отдельной командой после завершения pipeline.

## Архитектурная идея

Пользовательский поток остаётся единым:

`presentation -> workflow -> agent -> mcp`

Распределение ответственности:

- `presentation`
  Разбирает CLI-команды и форматирует результат.
- `workflow`
  Запускает прикладной сценарий и получает итоговый результат.
- `agent`
  Является единственной точкой orchestration, выбора tools и маршрутизации между серверами.
- `mcp`
  Предоставляет атомарные инструменты и не содержит пользовательского orchestration flow.

## Целевой multi-server flow

### Серверы

`stateless` сервер:

- публикует инструменты получения исходных данных;
- используется как источник публикаций для pipeline.

`stateful` сервер:

- публикует инструменты обработки, сохранения и чтения summary;
- используется как сервер состояния и persistence.

### Предпочтительный сценарий вызовов

Рекомендуемый поток для команды:

`tool summary posts 10 long`

Пошагово:

1. CLI parser создаёт workflow-команду.
2. Workflow запускает pipeline через agent.
3. Agent выбирает `list_posts` как source tool на `stateless` сервере.
4. Agent вызывает `list_posts` на `stateless` сервере.
5. Agent получает список публикаций и выбирает 3 записи по стратегии:
   - `long` — 3 самых длинных;
   - `short` — 3 самых коротких.
6. Agent выбирает `merge_posts` на `stateful` сервере.
7. Agent вызывает `merge_posts` на `stateful` сервере.
8. Agent выбирает `save_summary` на `stateful` сервере.
9. Agent вызывает `save_summary` на `stateful` сервере.
10. Workflow получает итоговый pipeline-result и возвращает его в CLI.

Проверочный follow-up flow:

1. Пользователь вызывает `tool summaries`.
2. Agent выбирает `list_saved_summaries` на `stateful` сервере.
3. Пользователь видит сохранённый результат.

Дополнительный follow-up flow:

1. Пользователь вызывает `tool summary saved summary-1`.
2. Agent выбирает `get_saved_summary` на `stateful` сервере.
3. Пользователь видит конкретную сохранённую запись.

## Требования к выбору инструментов

- Агент должен использовать capability snapshot подготовленных серверов, а не жёстко зашитое знание только о CLI-команде.
- Агент должен выбирать источник данных из tools, найденных на `stateless` сервере.
- Агент должен выбирать merge/save tools из tools, найденных на `stateful` сервере.
- Если необходимый tool недоступен, агент должен возвращать прикладную ошибку с понятным сообщением.
- Порядок вызовов должен быть детерминированным и проверяемым тестами.

## Требования к маршрутизации

- Пользователь не выбирает сервер вручную.
- Presentation-layer не знает, на каком endpoint находится конкретный tool.
- Workflow-layer не знает transport-детали и server endpoint.
- Agent-layer сам определяет:
  - какой tool нужен на текущем шаге;
  - на каком сервере он доступен;
  - какой endpoint использовать для вызова.

## Требования к данным и контрактам

### Источник данных

Для cross-server flow нужен инструмент-источник, доступный на `stateless` сервере.

Зафиксированный вариант для проекта:

- использовать существующий `list_posts` как source tool для pipeline;
- агент сам отбирает нужное количество записей из полученного списка;
- добавление отдельного нового source tool не планируется, если `list_posts` можно расширить до нужного structured contract без усложнения проекта.

Требования:

- `list_posts` должен возвращать structured content, пригодный для orchestration;
- agent не должен парсить человекочитаемый текст ради выполнения pipeline;
- данных должно быть достаточно для выбора 3 публикаций и последующего merge.

### Инструменты обработки и сохранения

На `stateful` сервере должны быть доступны:

- `merge_posts`
- `save_summary`
- `list_saved_summaries`
- `get_saved_summary`

Требования:

- `merge_posts` принимает уже выбранные публикации и не занимается их поиском;
- `save_summary` сохраняет итоговый draft и возвращает metadata;
- `list_saved_summaries` и `get_saved_summary` читают результат из локального хранилища;
- все инструменты возвращают structured content, пригодный для orchestration и тестов.

## Требования к длинному flow

- Один пользовательский запрос должен приводить к нескольким последовательным tool calls.
- Среди этих вызовов должны быть вызовы минимум на двух разных MCP-серверах.
- Между шагами должны передаваться реальные промежуточные данные:
  - публикации от source tool;
  - выбранные публикации после agent-side selection;
  - draft summary после merge;
  - saved summary metadata после save.
- Ошибка на любом шаге должна завершать flow с понятным сообщением и без маскировки причины.

## Нефункциональные требования

- Нельзя ломать существующие команды:
  - `tool posts`
  - `tool post <postId>`
  - `tool start-random-posts [intervalMinutes]`
  - `tool summaries`
  - `tool summary saved <summaryId>`
- Новый orchestration flow должен сохранять текущую архитектурную цепочку `presentation -> workflow -> agent -> mcp`.
- Решение должно оставаться минимальным и учебным, без лишних уровней абстракции.
- Для нового orchestration-кода и пользовательского сценария должна быть добавлена документация.
- E2E-проверка должна быть воспроизводимой локально.

## Этапы реализации

### Этап 1. Спецификация и фиксация контракта

Цель:

- зафиксировать целевой multi-server сценарий и границы ответственности слоёв.

Результат этапа:

- в `docs` есть отдельное ТЗ на multi-server orchestration;
- сценарий и критерии готовности зафиксированы письменно;
- определено, какой tool считается источником данных и какие tools остаются на stateful server.

Шаги этапа:

1. Зафиксировать целевой cross-server flow.
2. Зафиксировать expected order вызовов.
3. Зафиксировать, какие tools и на каких серверах участвуют в сценарии.
4. Зафиксировать `list_posts` как source tool и отметить требование к его structured response.
5. Зафиксировать формат пользовательского результата и ошибки.

### Этап 2. Контракты данных и capability discovery для flow

Цель:

- подготовить agent-level модель, в которой длинный flow строится по найденным capabilities, а не по жёсткой привязке к одному серверу.

Результат этапа:

- агент умеет валидировать наличие source, merge и save tools на разных серверах;
- подготовленный snapshot содержит достаточно данных для cross-server routing.

Шаги этапа:

1. Проверить, хватает ли существующих structured responses у `stateless` tools.
2. При необходимости доработать structured content у source tool.
3. Зафиксировать в agent-layer правила выбора tool по назначению:
   - source tool;
   - merge tool;
   - save tool;
   - read tools.
4. Убрать из длинного flow жёсткую зависимость только от `LOCAL_STATEFUL_MCP_SERVER` там, где она больше не оправдана.

### Этап 3. Реализация cross-server orchestration в agent/workflow

Цель:

- перенести основной pipeline в настоящий multi-server flow.

Результат этапа:

- одна workflow-команда запускает последовательность вызовов на разных серверах;
- agent выбирает и вызывает нужные инструменты в правильном порядке;
- результат остаётся единым и удобным для CLI.

Шаги этапа:

1. Обновить pipeline в `DefaultAgent`.
2. Реализовать поиск source tool на `stateless` сервере.
3. Реализовать agent-side selection трёх публикаций.
4. Реализовать поиск и вызов `merge_posts` на `stateful` сервере.
5. Реализовать поиск и вызов `save_summary` на `stateful` сервере.
6. Вернуть единый `summary_pipeline` result.

### Этап 4. Обновление routing-модели и command definitions

Цель:

- сделать пользовательский сценарий согласованным с capability-based routing.

Результат этапа:

- user-facing команда доступна только если найден полный набор нужных capabilities;
- обычные single-tool команды продолжают работать как раньше;
- новая команда демонстрирует routing не только на фиксированный сервер, но и на orchestration flow.

Шаги этапа:

1. Пересмотреть `ToolSummaryPostsAgentCommandDefinition`.
2. Определить, как команда `tool summary posts` считается доступной:
   - есть source tool на `stateless`;
   - есть merge/save tools на `stateful`.
3. При необходимости расширить routing-модель или resolver так, чтобы availability вычислялась по нескольким серверам.
4. Сохранить обратную совместимость остальных command definitions.

### Этап 5. Тесты на выбор и порядок вызовов

Цель:

- доказать, что агент выбирает корректные инструменты и вызывает их в правильной последовательности.

Результат этапа:

- есть unit-тесты на cross-server orchestration;
- есть проверки порядка вызовов;
- есть проверки частичных ошибок и отсутствующих tools.

Шаги этапа:

1. Добавить agent-level тест, где source tool находится на одном сервере, а merge/save на другом.
2. Проверить порядок вызовов:
   - source
   - merge
   - save
3. Проверить, что в merge передаются только выбранные агентом публикации.
4. Проверить, что ошибка source/merge/save шага корректно прерывает flow.
5. Проверить, что команда считается недоступной, если не хватает хотя бы одной обязательной capability.

### Этап 6. Сквозная e2e-проверка

Цель:

- подтвердить реальную работу сценария на локально поднятых MCP-серверах.

Результат этапа:

- scripted e2e-проверка воспроизводит multi-server pipeline;
- проверка подтверждает сохранение и последующее чтение summary.

Шаги этапа:

1. Обновить `scripts/check-e2e.ps1`.
2. Включить в e2e последовательность:
   - запуск `stateless` и `stateful` серверов;

   - выполнение `tool summaries`;
   - выполнение `tool summary saved <id>` при необходимости.
3. Проверить наличие ожидаемых фрагментов в выводе.
4. Проверить создание и чтение локального storage.

### Этап 7. Документация и финальная полировка

## Сценарий проверки

Обязательный сценарий проверки:

1. Запустить `stateless` сервер.
2. Запустить `stateful` сервер.
3. Запустить клиент.
4. Выполнить `tool summary posts 10 long`.
5. Убедиться, что summary успешно сохранён.
6. Выполнить `tool summaries`.
7. Убедиться, что сохранённый summary отображается в списке.
8. Выполнить `tool summary saved summary-1` или другой фактический id.
9. Убедиться, что возвращается полная сохранённая запись.

Что должно быть подтверждено:

- в одном пользовательском сценарии используются tools с разных серверов;
- агент корректно выбирает и вызывает tools в нужном порядке;
- результат сохраняется и затем читается отдельной командой;
- flow работает end-to-end без ручного выбора сервера пользователем.

## Критерии готовности

Реализация считается завершённой, если:

- агент выполняет длинный flow как минимум через два MCP-сервера;
- source tool и tools обработки/сохранения действительно живут на разных серверах;
- порядок вызовов подтверждён unit-тестами;
- cross-server flow подтверждён e2e-проверкой;
- существующие команды не сломаны;
- README и `docs` отражают новое состояние проекта.
