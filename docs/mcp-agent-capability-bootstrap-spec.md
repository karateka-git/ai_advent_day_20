# Техническое задание

## Название

Автоподключение агента к MCP-серверам и публикация пользовательских команд на основе доступных возможностей.

## Цель

Перевести текущий CLI-сценарий от пользовательской модели:

- `connect`
- просмотр доступных инструментов
- вызов `tool ...`

к агентной модели, в которой:

1. агент при старте сам подключается к известным MCP-серверам;
2. агент получает от каждого сервера список доступных `tools`;
3. агент строит из найденных `tools` список поддерживаемых пользовательских команд;
4. пользователь видит только доступные команды и не управляет подключениями вручную;
5. вызов команды маршрутизируется агентом на подходящий MCP-сервер без участия пользователя.

## Проблема

Текущее состояние системы содержит несколько архитектурных и UX-противоречий:

- пользовательская команда `connect` раскрывает внутреннюю инфраструктурную механику, хотя подключением к MCP должен управлять агент;
- команды `tool ...` концептуально выглядят как доступные только после `connect`, но технически могут инициировать подключение под капотом;
- presentation-слой уже начал брать на себя проверку состояния подключения, что нарушает границу ответственности;
- пользователь вынужден думать о том, что существует сервер и что у него есть `tools`, хотя фактически он должен видеть только доступные сценарии приложения.

## Целевая модель

После реализации агент должен вести себя так:

1. При старте приложения агент начинает подготовку.
2. Агент подключается к заранее известным MCP-серверам.
3. Агент получает от них `serverInfo` и `tools/list`.
4. Агент сопоставляет известные MCP-инструменты с поддерживаемыми пользовательскими командами.
5. После завершения подготовки CLI показывает только реально доступные команды.
6. Пользователь вызывает команду, не зная:
   - на каком сервере находится нужный `tool`;
   - как выполнено подключение;
   - был ли выполнен discovery конкретного сервера.
7. Агент сам выбирает подходящий сервер и вызывает соответствующий `tool`.

## Что пользователь должен видеть

На старте:

- сообщение о подготовке агента;
- визуальное подтверждение того, что идёт загрузка;
- после завершения подготовки — приветственное сообщение и список/подсказку по доступным командам.

Пользователь не должен видеть:

- команду `connect`;
- необходимость вручную инициировать discovery серверов;
- информацию о том, на каком сервере находится конкретный `tool`;
- внутренние сообщения transport-уровня.

## Требования к UX

- Команда `connect` удаляется из пользовательского CLI-контракта.
- На старте до завершения подготовки не показывается финальное приветственное сообщение о готовности агента.
- Во время подготовки показывается сообщение `Подготовка агента...`.
- Сообщение подготовки должно визуально анимироваться точками, чтобы было видно, что загрузка продолжается.
- После завершения подготовки выводится итоговое приветственное сообщение и подсказка по доступным командам.
- Команда `help` показывает только реально доступные пользовательские команды.
- Если по итогам подготовки какой-то поддерживаемый сценарий недоступен, он не должен показываться пользователю как доступный.

## Функциональные требования

- Агент должен уметь автоматически подключаться к одному или нескольким MCP-серверам на старте.
- Список стартовых MCP-серверов должен задаваться в одном месте конфигурации приложения.
- После подключения к каждому серверу агент должен получить:
  - endpoint;
  - server info;
  - список `tools`.
- Агент должен хранить реестр подключённых MCP-серверов и их возможностей.
- Агент должен хранить индекс доступных пользовательских команд, построенный из найденных `tools`.
- При вызове пользовательской команды агент должен сам определить:
  - какой MCP-инструмент нужен;
  - на каком из подключённых серверов он доступен;
  - какой endpoint использовать для `tools/call`.
- Если для пользовательской команды не найден подходящий `tool`, агент должен вернуть прикладную ошибку верхним слоям.
- Presentation-слой не должен хранить состояние подключения к MCP-серверам.

## Архитектурные требования

### `presentation`

Должен:

- запускать подготовку агента при старте приложения;
- показывать пользователю состояние подготовки;
- показывать только пользовательские команды и результаты их выполнения.

Не должен:

- хранить флаг подключения к MCP;
- решать, доступна ли команда на основании прямого знания о сервере;
- знать, какой `tool` лежит на каком endpoint.

### `workflow`

Должен:

- работать с пользовательскими командами как с прикладными сценариями;
- делегировать подготовку и вызовы агенту.

Не должен:

- хранить карту серверов;
- заниматься discovery `tools/list`;
- знать transport-детали MCP.

### `agent`

Должен:

- быть единственной точкой знания о подключённых MCP-серверах;
- выполнять стартовый discovery возможностей;
- строить карту:
  - пользовательская команда;
  - соответствующий MCP `tool`;
  - сервер, на котором этот `tool` доступен;
- самостоятельно маршрутизировать вызов команды на нужный сервер.

Именно агент должен принимать решение о доступности пользовательской команды.

### `mcp`

Должен:

- предоставлять операции подключения и получения списка `tools`;
- предоставлять вызов `tools/call`;
- оставаться инфраструктурным слоем без прикладного знания о пользовательских командах.

## Ограничения и допущения

- На этом этапе пользователь не выбирает сервер вручную.
- На этом этапе не добавляются новые служебные команды вроде `status`, `refresh` и т.п.
- На этом этапе допустимо сохранить текущий набор известных пользовательских сценариев:
  - `tool posts`
  - `tool post <postId>`
- Но доступность этих сценариев должна определяться не жёстко в `App`, а результатом discovery у агента.
- Если несколько серверов публикуют один и тот же поддерживаемый `tool`, правило выбора сервера должно быть централизовано в агенте.
- Если сервер недоступен на старте, приложение не должно падать безусловно, если доступны другие полезные сценарии.

## Ожидаемые изменения в модели приложения

Текущее поведение:

- пользователь вручную вызывает `connect`;
- `connect` получает список инструментов;
- отдельная `tool`-команда может фактически подключиться сама под капотом.

Целевое поведение:

- приложение при старте готовит агента автоматически;
- агент сам собирает capabilities;
- пользователь взаимодействует только с прикладными командами;
- `connect` больше не участвует в пользовательском сценарии.

## Этапы реализации

### Этап 1. Формализация агентной модели возможностей

Цель:

- зафиксировать новую модель ответственности;
- описать, какие данные агент хранит после подготовки;
- зафиксировать, как пользовательские команды выводятся из MCP-возможностей.

Результат этапа:

- описана модель стартового discovery;
- описана модель хранения capabilities;
- описана связь между пользовательскими командами и MCP `tools`.

Шаги этапа:

1. Зафиксировать, что `connect` больше не является пользовательской командой.
2. Описать, какие серверы агент подключает при старте.
3. Описать, какие данные агент хранит по каждому серверу.
4. Описать, как из `tools/list` строится список доступных пользовательских команд.
5. Зафиксировать правило маршрутизации пользовательской команды на сервер и `tool`.

### Этап 2. Стартовая подготовка агента

Цель:

- внедрить lifecycle-сценарий, в котором агент при старте автоматически получает возможности MCP-серверов.

Результат этапа:

- у агента появляется явный сценарий подготовки;
- результат подготовки содержит доступные команды и данные по подключённым серверам;
- верхние слои больше не требуют пользовательского `connect`.

Шаги этапа:

1. Добавить в агент новый сценарий подготовки/инициализации.
2. Подключить к нему список известных MCP-серверов.
3. Получать по каждому серверу `serverInfo` и `tools/list`.
4. Сохранять итог подготовки в agent-level состоянии или реестре.
5. Добавить локальные тесты на успешную и частично неуспешную подготовку.

### Этап 3. Реестр возможностей и маршрутизация команд

Цель:

- научить агента принимать пользовательскую команду и выбирать подходящий MCP `tool` и сервер автоматически.

Результат этапа:

- агент хранит индекс доступных возможностей;
- пользовательская команда больше не жёстко привязана к endpoint в presentation;
- вызов маршрутизируется через agent-level capability registry.

Шаги этапа:

1. Ввести agent-level модель реестра подключённых серверов.
2. Ввести agent-level модель доступной пользовательской команды.
3. Построить mapping между поддерживаемыми пользовательскими командами и найденными `tools`.
4. Перенести выбор endpoint/tool в агент.
5. Добавить unit-тесты на маршрутизацию и ошибки отсутствующих возможностей.

### Этап 4. Перестройка workflow и presentation под новую модель

Цель:

- убрать из верхних слоёв пользовательский `connect`;
- сделать CLI зависимым от уже подготовленных агентом возможностей.

Результат этапа:

- `connect` убран из parser/help/startup UX;
- presentation не хранит состояние подключения;
- `help` и стартовые подсказки строятся из данных агента.

Шаги этапа:

1. Удалить пользовательскую команду `connect` из CLI-контракта.
2. Обновить parser и help под новый набор команд.
3. Передавать в presentation итог уже подготовленного состояния агента.
4. Убрать любые presentation-level проверки подключения к MCP.
5. Добавить unit-тесты на новый CLI-контракт.

### Этап 5. UX подготовки агента

Цель:

- сделать старт приложения понятным для пользователя, пока агент подключается к серверам и получает возможности.

Результат этапа:

- на старте показывается состояние подготовки;
- пользователь не получает сообщение о готовности до завершения discovery;
- появляется анимация загрузки с точками.

Шаги этапа:

1. Определить точку запуска подготовки агента до старта интерактивного цикла.
2. Добавить вывод `Подготовка агента...`.
3. Реализовать простую анимацию точек без шума в консоли.
4. После завершения подготовки показать итоговое приветственное сообщение.
5. Проверить UX вручную через проектный сценарий сборки/запуска.

### Этап 6. Сквозная проверка и синхронизация документации

Цель:

- подтвердить, что новая агентная модель действительно работает end-to-end и понятна из документации.

Результат этапа:

- ручной и scripted сценарии отражают новую модель без `connect`;
- документация синхронизирована с фактическим UX.

Шаги этапа:

1. Обновить сценарии ручной проверки.
2. Обновить scripted smoke/e2e-проверки под отсутствие `connect`.
3. Обновить README и связанные docs.
4. Зафиксировать результаты этапов в implementation log.

## Критерии готовности

Реализация считается завершённой, если:

- пользовательская команда `connect` исчезла из основного CLI-сценария;
- агент на старте автоматически получает возможности известных MCP-серверов;
- пользователь видит только реально доступные команды;
- agent-layer сам выбирает, на каком сервере вызывать нужный `tool`;
- presentation-слой не хранит состояние подключения к MCP;
- на старте показывается понятное сообщение о подготовке агента;
- после завершения подготовки пользователь получает финальное сообщение о готовности и доступных командах;
- сценарий воспроизводим при ручной проверке через проектную команду сборки.
