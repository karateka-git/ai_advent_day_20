# Техническое Задание

## Название

Рефакторинг клиентской части в агентно-ориентированную архитектуру с отделением `application`, `agent`, `mcp` и `presentation`.

## Цель

Преобразовать текущую клиентскую часть проекта из прямого CLI-driven MCP client в расширяемую архитектуру, где:

1. пользовательский ввод обрабатывается через слой `presentation`;
2. слой `application` принимает команды и управляет сценарием выполнения;
3. слой `agent` выступает посредником между application-командами и MCP;
4. слой `mcp` инкапсулирует техническую работу с MCP SDK;
5. UI не зависит от MCP SDK и может быть в будущем заменён на Web, Android или другой интерфейс;
6. архитектура остаётся готовой к будущему расширению в сторону LLM-orchestration, но без внедрения LLM на текущем этапе.

## Контекст

Сейчас проект уже умеет:

- поднимать локальный MCP server;
- подключать к нему client;
- выполнять `initialize -> tools/list`;
- выводить результат в CLI;
- проходить воспроизводимую e2e-проверку.

Текущая реализация решает базовую задачу, но ещё недостаточно разделяет:

- пользовательский ввод;
- сценарии выполнения команд;
- агентную оркестрацию;
- технический доступ к MCP;
- форматирование результата под конкретный UI.

Следующий шаг проекта — ввести устойчивую архитектуру, где агент присутствует в системе уже сейчас, а UI перестаёт быть центральной точкой исполнения логики.

## Целевой Архитектурный Принцип

Система должна быть организована вокруг следующей цепочки:

`presentation -> application -> agent -> mcp`

Расшифровка:

- `presentation` отвечает только за приём пользовательского ввода и отображение результата;
- `application` знает, какие команды поддерживаются и какой сценарий нужно выполнить;
- `agent` знает, как выполнить команду через взаимодействие с MCP;
- `mcp` знает только технический протокол общения с MCP server.

### Важное Ограничение

На текущем этапе агент:

- не взаимодействует с LLM;
- не принимает самостоятельных решений о выборе tool;
- не строит reasoning loop;
- не занимается автопланированием.

Но его API и ответственность должны закладываться так, чтобы в будущем этот слой можно было расширить до более сложного orchestration-сценария.

## Целевая Структура Пакетов

### `application/command`

Содержит application-команды, не зависящие от CLI, Web, Android и других UI.

Минимальный состав:

- `Command`
- `ConnectCommand`

Назначение:

- формализовать, что именно просит выполнить внешний слой;
- не привязывать команду к конкретному представлению ввода.

### `application/result`

Содержит результаты выполнения команд приложения.

Минимальный состав:

- `CommandResult`
- `ConnectResult`

Назначение:

- возвращать структурированный результат;
- не смешивать сценарий выполнения с конкретным текстом для консоли;
- оставить возможность разного отображения результата в CLI, Web и Android.

### `application/service`

Содержит orchestration application-уровня.

Минимальный состав:

- `ApplicationCommandHandler`

Назначение:

- принимать `Command`;
- делегировать выполнение в `agent`;
- преобразовывать агентный результат в `CommandResult`.

Слой `application` не должен знать деталей MCP SDK.

### `agent`

Содержит агентный слой, который стоит между `application` и `mcp`.

Минимальный состав:

- `Agent`
- `AgentRequest`
- `AgentResponse`

Назначение:

- принимать запрос от application;
- выполнять агентный сценарий;
- обращаться к `McpClient`;
- возвращать структурированный результат.

На текущем этапе агент должен уметь:

- выполнить сценарий `connect`;
- запросить у MCP server server info;
- получить список tools;
- подготовить ответ для application.

В будущем этот же слой должен быть удобен для расширения в сторону:

- `resources/list`;
- `tools/call`;
- многошаговых цепочек;
- LLM-driven выбора действий.

### `mcp`

Содержит интеграцию с MCP.

Минимальный состав:

- `McpClient`
- `DefaultMcpClient`
- `model/...`

Назначение:

- изолировать Kotlin MCP SDK в одном месте;
- дать остальным слоям понятный проектный API;
- не допускать протекания SDK-типов в `application`, `agent` и `presentation`.

`McpClient` на текущем этапе должен поддерживать:

- подключение к серверу;
- получение server info;
- получение `tools/list`.

Проектные модели в `mcp/model` должны быть подготовлены под будущее расширение, чтобы позже туда можно было добавить:

- ресурсы;
- tool call result;
- структурированные ошибки MCP.

### `presentation/cli`

Содержит CLI-адаптер.

Минимальный состав:

- `CliCommandParser`
- `CliOutputFormatter`

Назначение:

- переводить CLI-аргументы в `application/command`;
- переводить `CommandResult` в консольный текст.

Этот слой не должен:

- знать детали Kotlin MCP SDK;
- выполнять MCP-запросы напрямую;
- хранить orchestration-логику команды.

## Функциональные Требования

### Обязательные Требования Для Первой Реализации

- Пользовательская команда `connect` должна по-прежнему работать.
- Внешнее поведение текущих проектных команд должно остаться совместимым:
  - `runClient` должен продолжать приводить к подключению и выводу tools list.
- После рефакторинга система должна по-прежнему:
  - подключаться к MCP server;
  - выполнять `initialize`;
  - получать `tools/list`;
  - выводить серверную информацию и список инструментов.
- Новые слои должны быть введены реально, а не только номинально через переименование файлов.

### Требования К Командам Application-Слоя

- `Command` должен быть общей абстракцией application-команд.
- `ConnectCommand` должен быть первой конкретной командой.
- `ApplicationCommandHandler` должен принимать `Command`, а не CLI-строку.
- `ApplicationCommandHandler` не должен зависеть от CLI parser-а.

### Требования К Агентному Слою

- Агент должен иметь собственные request/response модели.
- Агент должен быть отдельным слоем, а не просто переименованным helper-методом.
- Агент должен использовать `McpClient`, а не работать с MCP SDK напрямую.
- Агент не должен форматировать CLI-строки.

### Требования К MCP-Слою

- `McpClient` должен быть проектным интерфейсом.
- `DefaultMcpClient` должен быть реализацией через текущий Kotlin MCP SDK.
- MCP SDK-типы должны быть локализованы в `mcp` слое настолько, насколько это разумно на текущем этапе.
- Приоритет — понятный и проектно-ориентированный API, а не тонкая обёртка “один-в-один” над SDK.

### Требования К Presentation-Слою

- `CliCommandParser` должен разбирать CLI-ввод в application-команды.
- `CliOutputFormatter` должен форматировать application-результаты.
- CLI слой должен остаться тонким адаптером.

## Нефункциональные Требования

- Архитектура должна быть понятной и не перегруженной избыточными абстракциями.
- Названия сущностей должны отражать реальные роли слоёв.
- Код должен быть пригоден для будущего появления других UI без переписывания `application`, `agent` и `mcp`.
- Код должен быть пригоден для будущего появления LLM в слое `agent` без перелома границ между слоями.
- Документация должна быть обновлена синхронно с кодом.
- KDoc/Javadoc для нового кода писать на русском.

## Явные Границы Ответственности

### Что Не Должно Быть В `presentation`

- прямых вызовов MCP SDK;
- знания о transport;
- знания о порядке `initialize -> tools/list`;
- сценарной логики выполнения команды.

### Что Не Должно Быть В `application`

- CLI-specific строк;
- форматирования под консоль;
- прямых вызовов MCP SDK.

### Что Не Должно Быть В `agent`

- парсинга CLI-команд;
- форматирования вывода под UI;
- привязки к PowerShell, Web, Android или другому интерфейсу.

### Что Не Должно Быть В `mcp`

- знания о пользовательских командах `connect`, `tools`, `resources`;
- знания о presentation-слое;
- знания об application-результатах.

## Целевые Контракты Первого Приближения

### `Command`

Общая абстракция application-команды.

Минимальная задача:

- дать единый тип команд, который может приходить из разных UI.

### `ConnectCommand`

Команда подключения к MCP и запроса базовой информации.

Минимальная задача:

- представить пользовательское намерение “подключиться и получить текущую MCP-информацию”.

### `CommandResult`

Общая абстракция результата выполнения application-команды.

### `ConnectResult`

Результат успешного или неуспешного выполнения `ConnectCommand`.

На текущем этапе должен содержать достаточно данных, чтобы presentation-слой мог вывести:

- endpoint;
- server info;
- список tools;
- информацию об ошибке, если сценарий завершился неуспешно.

### `AgentRequest`

Нормализованный запрос от application к агенту.

### `AgentResponse`

Нормализованный ответ агента в application-слой.

### `McpClient`

Проектный контракт доступа к MCP.

На текущем этапе должен содержать методы, достаточные для сценария `connect`.

## Критерии Готовности

Рефакторинг считается успешно завершённым, если:

- в проекте реально присутствуют слои `application`, `agent`, `mcp`, `presentation`;
- команда `connect` работает через полную новую цепочку;
- CLI больше не зависит напрямую от MCP SDK;
- agent находится между application и mcp;
- проект сохраняет текущий рабочий сценарий `initialize -> tools/list`;
- e2e-проверка остаётся зелёной;
- архитектура понятна как основа для будущих UI и будущего agent/LLM-расширения.

## Что Не Входит В Этот Объём

На текущем этапе в ТЗ не входит:

- внедрение LLM;
- автоматический выбор tool через модель;
- полноценный reasoning loop;
- Web UI;
- Android UI;
- вызовы `resources/list`, если они пока ещё не встроены в первый рабочий вертикальный срез;
- полноценный `tools/call` orchestration поверх нескольких agent-решений.

Это не исключается из будущего развития, но не должно размывать текущий рефакторинг.
