## Этап 2. Минимальный MCP Server

Статус: завершён

Цель этапа:

- поднять локальный MCP server по HTTP transport;
- зарегистрировать минимальные инструменты `ping` и `echo`;
- подготовить серверный bootstrap для следующего этапа с клиентом.

Выполненные действия:

1. Шаги этапа 2 зафиксированы в ТЗ.
2. Добавлен локальный Ktor/CIO bootstrap для MCP server с endpoint `/mcp`.
3. Добавлена фабрика `createMcpServer()` с `ServerCapabilities.tools`.
4. Зарегистрированы инструменты `ping` и `echo`.
5. Добавлен тест на регистрацию инструментов в server registry.
6. Выполнены `.\gradlew.bat build` и `.\gradlew.bat installDist`.
7. Отдельно проверен запуск серверного entrypoint: сервер поднимается и в логах видна регистрация инструментов.

Принятые решения:

- использовать `Application.mcpStreamableHttp(...)` из официального Kotlin SDK;
- держать bootstrap сервера отдельно от фабрики MCP server;
- использовать публичный `server.tools` как простой способ подтвердить регистрацию инструментов на этом этапе.

Проверка:

- `.\gradlew.bat build` завершается успешно;
- `createMcpServer().tools.keys` содержит `ping` и `echo`;
- серверный процесс стартует локально и логирует запуск на `http://127.0.0.1:3000/mcp`.

Наблюдение:

- raw проверка через `curl` пока возвращает `406 Not Acceptable`, хотя сервер стартует и создаёт session;
- этот нюанс нужно перепроверить на `Этапе 3` через официальный MCP client, чтобы не смешивать реализацию сервера и раннюю отладку клиента.

## Этап 3. Минимальный MCP Client

Статус: завершён

Цель этапа:

- реализовать клиента, который подключается к локальному endpoint сервера;
- выполнить `initialize`;
- запросить `tools/list`;
- вывести результат в консоль.

Выполненные действия:

1. Шаги `Этапа 3` зафиксированы в ТЗ.
2. Реализован клиентский entrypoint на официальном Kotlin SDK с `StreamableHttpClientTransport`.
3. Добавлен вывод серверной информации после `initialize` и форматированный список инструментов после `tools/list`.
4. Для клиентской части добавлены локальные unit-проверки на форматирование консольного вывода.
5. Во время живой проверки обнаружена причина прежнего `406 Not Acceptable`: на сервере отсутствовала установленная JSON-сериализация для ответов transport-уровня.
6. Серверный bootstrap скорректирован:
   - подключён `ContentNegotiation` c `json()`;
   - выбран stateless Streamable HTTP transport как более простой и устойчивый для текущего сценария `initialize -> tools/list`.
7. Выполнена живая сквозная проверка через отдельный запуск `runServer` и `runClient` с захватом stdout/stderr:
   - клиент подключается к `http://127.0.0.1:3000/mcp`;
   - получает `serverInfo`;
   - выводит инструменты `ping` и `echo`.

Принятые решения:

- использовать официальный Kotlin MCP client API без самописного JSON-RPC слоя;
- оставить транспорт HTTP-based, но на текущем этапе перейти на stateless Streamable HTTP, так как для минимального reference-сценария он проще и не требует отдельного session lifecycle;
- не оставлять в кодовой базе нестабильный интеграционный тест, если тот зависит от капризного lifecycle фоновых Gradle/Ktor-процессов; вместо этого фиксировать проверенный end-to-end сценарий в журнале и держать зелёные unit-тесты.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- `runClient` при живом `runServer` выводит:
  - подключение к локальному endpoint;
  - `local_mcp_server` как имя сервера;
  - список инструментов `ping` и `echo`;
- для stateless transport клиентский SDK логирует ожидаемое сообщение `Server returned 405 for GET/SSE, stream disabled.`, что в данном режиме является нормальным поведением.

Коммиты этапа:

- `4faef67` — реализация Streamable HTTP MCP client;

Принятые решения:
