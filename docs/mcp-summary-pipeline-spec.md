# Техническое задание

## Название

Автоматический pipeline композиции MCP-инструментов для сборки и сохранения summary по случайным публикациям.

## Цель

Развить текущий sandbox-проект до сценария, в котором одна пользовательская команда автоматически запускает цепочку из нескольких MCP-инструментов:

1. первый инструмент получает `n` случайных публикаций;
2. агент по внутренней логике выбирает 3 публикации по стратегии `long` или `short`;
3. второй инструмент объединяет выбранные публикации в один summary;
4. третий инструмент сохраняет summary в локальное хранилище;
5. дополнительный инструмент позволяет получить все сохранённые summary.

Минимально успешный результат:

- пользователь вызывает одну CLI-команду;
- `workflow -> agent -> mcp` автоматически выполняют цепочку шагов;
- данные корректно передаются между шагами;
- итоговый summary сохраняется и затем доступен отдельной командой чтения хранилища.

## Контекст

Проект уже умеет:

- поднимать локальный `stateless` MCP server;
- поднимать отдельный `stateful` MCP server;
- вызывать `list_posts`, `fetch_post` и `start_random_posts`;
- маршрутизировать пользовательские команды через цепочку `presentation -> workflow -> agent -> mcp`;
- выполнять scripted e2e-проверку через PowerShell.

Следующий естественный шаг — показать не одиночный `tools/call`, а композицию нескольких MCP-инструментов в одном пользовательском сценарии.

## Пользовательский сценарий

Основная команда:

- `tool summary posts <count> [strategy]`

Примеры:

- `tool summary posts 10`
- `tool summary posts 10 long`
- `tool summary posts 10 short`

Поведение:

- `<count>` — количество случайных публикаций, которые нужно получить первым инструментом;
- `[strategy]` — стратегия выбора трёх публикаций агентом;
- если стратегия не указана, используется `long`.

Команда просмотра сохранённых summary:

- `tool summaries`

## Pipeline

Пайплайн реализуется не внутри одного MCP-инструмента, а в orchestration-слое приложения:

`presentation -> workflow -> agent -> mcp`

Поток выполнения основной команды:

1. CLI parser преобразует `tool summary posts <count> [strategy]` в workflow-команду.
2. Workflow запускает pipeline.
3. Агент вызывает MCP-инструмент `pick_random_posts`.
4. Агент по внутренней логике выбирает 3 публикации:
   - `long` — 3 публикации с самым длинным `body`;
   - `short` — 3 публикации с самым коротким `body`.
5. Агент вызывает MCP-инструмент `merge_posts`.
6. Агент вызывает MCP-инструмент `save_summary`.
7. Workflow возвращает пользователю итог выполнения pipeline.

Поток выполнения команды просмотра:

1. CLI parser преобразует `tool summaries` в workflow-команду.
2. Workflow делегирует вызов агенту.
3. Агент вызывает MCP-инструмент `list_saved_summaries`.
4. Результат выводится в CLI.

## Контракты MCP-инструментов

### `pick_random_posts`

Назначение:

- получить случайные публикации из локального русскоязычного mock-каталога для начала pipeline.

Вход:

- `count: Int`

Требования:

- инструмент должен брать данные из локального каталога тестовых публикаций, встроенного в проект;
- если `count < 1`, должен возвращаться понятный error-result;
- результат должен быть структурированным, чтобы агент мог выбрать 3 публикации без парсинга текстового вывода.

Выход:

- список публикаций с полями:
  - `id`
  - `userId`
  - `title`
  - `body`

### `merge_posts`

Назначение:

- объединить 3 выбранные публикации в единый summary.

Вход:

- `posts: List<Post>`
- `strategy: String`

Требования:

- инструмент не выбирает публикации сам;
- инструмент только формирует итоговый summary на основе уже выбранных 3 публикаций;
- результат должен быть человекочитаемым.

Выход:

- summary-объект с полями:
  - `title`
  - `content`
  - `sourcePostIds`
  - `strategy`

### `save_summary`

Назначение:

- сохранить summary в локальное хранилище.

Вход:

- `title: String`
- `content: String`
- `sourcePostIds: List<Int>`
- `strategy: String`

Требования:

- summary должен сохраняться в локальное проектное хранилище;
- хранилище должно быть воспроизводимо и не требовать внешней базы;
- сохранение должно возвращать идентификатор и метаданные.

Выход:

- `summaryId`
- `savedAt`
- `sourcePostIds`
- `strategy`

### `list_saved_summaries`

Назначение:

- вернуть все сохранённые summary из локального хранилища.

Вход:

- без аргументов

Выход:

- список сохранённых summary с метаданными и содержимым.

## Локальное хранилище

Для текущего этапа используется простое файловое хранилище внутри проекта.

Требования:

- хранилище должно быть локальным и понятным для ручной проверки;
- формат должен быть удобен для автотестов;
- структура должна позволять хранить несколько summary.

Рекомендуемый вариант:

- JSON-файл в `build/tmp` или другой локальной директории проекта;
- сериализация списка summary или append-модель через проектные serializable-модели.

Фактический вариант реализации:

- локальный JSON storage размещается в `build/tmp/mcp-summary-storage/saved-summaries.json`;
- первый шаг pipeline использует встроенный набор русскоязычных публикаций, чтобы e2e-проверка не зависела от внешней сети.

## Функциональные требования

- На `stateful` MCP server должны появиться новые инструменты:
  - `pick_random_posts`
  - `merge_posts`
  - `save_summary`
  - `list_saved_summaries`
- Основной pipeline должен запускаться одной CLI-командой `tool summary posts <count> [strategy]`.
- Выбор 3 публикаций не должен быть отдельным MCP-инструментом.
- Выбор 3 публикаций должен выполняться агентом по внутренней логике.
- Должна быть отдельная команда `tool summaries` для чтения локального хранилища.
- Данные между шагами pipeline должны передаваться автоматически без ручного ввода пользователя.

## Нефункциональные требования

- Нельзя ломать существующие команды `tool posts`, `tool post <postId>` и `tool start-random-posts [intervalMinutes]`.
- Новый pipeline не должен обходить агентный слой.
- `presentation`-слой не должен знать детали MCP SDK.
- `workflow`-слой не должен знать детали transport и SDK.
- Локальное хранилище должно быть воспроизводимым и пригодным для тестов.
- Для нового orchestration-кода и публичного сценария должна быть добавлена документация.

## Архитектурные ограничения

### `presentation`

Должен:

- разобрать новые команды пользователя;
- создать соответствующие workflow-команды;
- отформатировать результат pipeline и чтения хранилища.

Не должен:

- выполнять шаги pipeline напрямую;
- вызывать MCP SDK напрямую.

### `workflow`

Должен:

- запускать pipeline одной командой;
- обрабатывать успех и ошибку на каждом шаге;
- возвращать presentation-friendly результат.

Не должен:

- знать детали JSON-схем MCP SDK;
- знать детали HTTP transport.

### `agent`

Должен:

- вызывать MCP-инструменты;
- реализовывать внутреннюю логику выбора 3 публикаций;
- передавать промежуточные данные между шагами pipeline.

### `mcp`

Должен:

- предоставлять атомарные инструменты;
- инкапсулировать детали SDK и server-side обработчиков;
- не брать на себя orchestration пользовательского pipeline.
