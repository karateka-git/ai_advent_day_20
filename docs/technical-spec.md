# Техническое задание

## Название

Минимальный MCP reference-проект для подключения клиента к локальному серверу и получения списка инструментов.

## Цель

Собрать маленький расширяемый sandbox-проект, который демонстрирует базовый lifecycle работы с MCP:

1. запуск локального MCP server;
2. подключение MCP client;
3. выполнение `initialize`;
4. выполнение `tools/list`;
5. вывод списка доступных инструментов в консоль.

## Контекст

Проект создаётся как учебный reference, но с прицелом на дальнейшее развитие. Поэтому решение должно быть достаточно простым для быстрого старта и при этом не мешать в будущем:

- выносить сервер наружу;
- заменять локальный endpoint на удалённый;
- добавлять новые инструменты и сценарии проверки.

Базовый технологический выбор текущего этапа:

- основной стек реализации: Kotlin;
- сервер и клиент по возможности держать в одном JVM-проекте, чтобы не усложнять старт и упростить дальнейшую интеграцию.

## Функциональные требования

- В проекте должен быть собственный MCP server.
- Сервер должен подниматься локально и предоставлять MCP endpoint по HTTP-based transport.
- Сервер должен объявлять минимум 1-2 простых инструмента, например `ping` и `echo`.
- В проекте должен быть собственный MCP client.
- Клиент должен подключаться к локальному MCP endpoint сервера.
- Клиент должен проходить базовый handshake MCP и выполнять `initialize`.
- После инициализации клиент должен вызывать `tools/list`.
- Клиент должен печатать в консоль список доступных инструментов в читаемом виде.

## Нефункциональные требования

- Код должен быть организован так, чтобы transport-логика не была жёстко смешана с остальной логикой примера.
- Endpoint сервера должен быть вынесен в конфигурацию, константу верхнего уровня или другой явно редактируемый параметр.
- Решение должно оставаться минимальным: без лишних слоёв абстракции, которые не нужны для базового сценария.
- Для нового публичного кода и важного orchestration-кода должна быть добавлена документация.

## Ожидаемая структура решения

- Модуль или пакет сервера с минимальной регистрацией MCP tools.
- Модуль или пакет клиента, который выполняет подключение и `tools/list`.
- Точка входа или команды запуска для сервера и клиента.
- README с краткой инструкцией по запуску.

## Этапы Реализации

### Этап 1. Инициализация Kotlin-проекта

Цель:

- развернуть минимальный Kotlin/JVM-проект;
- подключить систему сборки и зависимости;
- подготовить базовую структуру каталогов для сервера, клиента и общей конфигурации.

Результат этапа:

- проект собирается;
- есть базовая структура исходников;
- добавлены зависимости, необходимые для Kotlin и MCP SDK.

Шаги этапа:

1. Зафиксировать технические решения для старта:
   - Gradle с Kotlin DSL;
   - Kotlin/JVM как основной target;
   - официальный `modelcontextprotocol/kotlin-sdk`;
   - отдельные артефакты `kotlin-sdk-client` и `kotlin-sdk-server`;
   - Ktor engine-зависимости для клиента и сервера;
   - Streamable HTTP как основной transport.
2. Развернуть Gradle-каркас проекта и базовые настройки сборки.
3. Добавить зависимости и минимальную структуру пакетов для сервера, клиента и конфигурации.
4. Убедиться, что проект успешно собирается на базовом каркасе.

### Этап 2. Минимальный MCP Server

Цель:

- поднять локальный MCP server по HTTP-based transport;
- зарегистрировать минимум 1-2 простых инструмента, например `ping` и `echo`.

Результат этапа:

- сервер стартует локально;
- сервер слушает выбранный endpoint;
- `tools/list` со стороны сервера возвращает зарегистрированные инструменты.

Шаги этапа:

1. Зафиксировать пошаговый план реализации сервера в ТЗ.
2. Реализовать локальный HTTP bootstrap для MCP server на Kotlin/Ktor.
3. Добавить минимальную конфигурацию MCP server и зарегистрировать инструменты `ping` и `echo`.
4. Проверить локальный запуск сервера и убедиться, что endpoint поднимается без ошибок.

### Этап 3. Минимальный MCP Client

Цель:

- реализовать клиента, который подключается к локальному endpoint сервера;
- выполнить `initialize`;
- запросить `tools/list`;
- вывести результат в консоль.

Результат этапа:

- клиент успешно соединяется с сервером;
- клиент проходит `initialize`;
- клиент печатает список доступных инструментов.

Шаги этапа:

1. Зафиксировать пошаговый план реализации клиента в ТЗ.
2. Подключить Streamable HTTP transport на стороне клиента через официальный Kotlin SDK.
3. Реализовать клиентский entrypoint, который:
   - подключается к локальному endpoint сервера;
   - проходит базовый handshake и `initialize`;
   - вызывает `tools/list`;
   - печатает серверную информацию и список доступных инструментов в читаемом виде.
4. Добавить локальную проверку клиентской части и убедиться, что проект продолжает успешно собираться.
5. Проверить запуск клиента против локального сервера и зафиксировать результат в журнале реализации.

### Этап 4. Сквозная Проверка

Цель:

- проверить полный сценарий `server -> client -> initialize -> tools/list`;
- убедиться, что решение работает end-to-end без ручных допущений.

Результат этапа:

- подтверждён рабочий сценарий запуска;
- зафиксирован ожидаемый консольный вывод или его формат;
- выявленные проблемы запуска устранены.

### Этап 5. Документация И Полировка

Цель:

- оформить README и проектные команды;
- зафиксировать, как запускать сервер и клиента;
- кратко описать, почему выбран именно HTTP-based transport.

Результат этапа:

- README отражает фактический способ запуска;
- `agent-preflight` обновлён под реальные команды проекта;
- проект готов как учебный reference для следующих задач.

## Сценарий проверки

1. Запустить локальный MCP server.
2. Запустить MCP client.
3. Убедиться, что соединение установлено без ошибок.
4. Убедиться, что `initialize` проходит успешно.
5. Убедиться, что `tools/list` возвращает зарегистрированные инструменты.
6. Убедиться, что клиент выводит их список в консоль.

## Критерии готовности

- есть отдельный sandbox-проект;
- в проекте реализован локальный MCP server;
- сервер работает по HTTP-based transport;
- клиент успешно подключается к серверу;
- клиент получает и выводит список инструментов;
- для переноса сервера на удалённый хост достаточно в основном поменять URL и инфраструктурное окружение, а не переписывать принцип подключения клиента.

## Ограничения и ориентиры

- На текущем этапе не требуется полноценная бизнес-логика.
- На текущем этапе не требуется сложная авторизация, хранение данных или production-hardening.
- Если стек ещё не выбран окончательно, предпочтение отдаётся варианту, который быстрее даст рабочий end-to-end результат без потери расширяемости.

## Результат этапа

Результатом считается код и документация, с помощью которых можно локально поднять MCP server, подключить к нему client и увидеть список доступных инструментов.
