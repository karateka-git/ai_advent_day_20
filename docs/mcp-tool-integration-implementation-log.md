## Этап 1. Выбор Инструмента И Контракта

Статус: завершён

Цель этапа:

- определить первый прикладной tool;
- зафиксировать источник данных, входной контракт и пользовательскую команду;
- подготовить письменную основу для следующих этапов реализации.

Выполненные действия:

1. Для `Этапа 1` в `docs/mcp-tool-integration-spec.md` добавлены подэтапы внедрения, чтобы разделить:
   - детализацию этапа;
   - выбор первого сценария;
   - описание контракта;
   - синхронизацию верхнеуровневой документации.
2. В отдельном документе `docs/mcp-tool-integration-contract.md` зафиксирован первый vertical slice для интеграции инструмента.
3. В качестве первого внешнего источника данных выбран публичный mock API `JSONPlaceholder`.
4. В качестве первого MCP-инструмента выбран `fetch_post`.
5. Для инструмента зафиксирован обязательный входной параметр `postId`.
6. В качестве первой пользовательской команды приложения зафиксирована команда `tool post <postId>`.
7. README и `MemoryBank/notes/overview.md` синхронизированы с выбранным следующим сценарием развития.

Принятые решения:

- начать с HTTP mock API, а не с Git или другой интеграции, чтобы минимизировать инфраструктурную сложность первого `tools/call`;
- выбрать сценарий чтения публикации как самый простой и воспроизводимый пример прикладного MCP-инструмента;
- фиксировать результат этапа не в пошаговых заметках, а в одном implementation log по аналогии с `mcp-reference-implementation-log.md`.

Проверка:

- в проекте есть отдельный контракт первого инструмента: `docs/mcp-tool-integration-contract.md`;
- верхнеуровневая документация теперь ссылается на активное ТЗ и на контракт первого инструмента;
- следующий этап реализации может опираться на письменный контракт, а не на устные договорённости.

## Этап 2. Реализация Tool На Сервере

Статус: завершён

Цель этапа:

- добавить на MCP server новый прикладной инструмент `fetch_post`;
- подключить к нему внешний mock API;
- подтвердить локальными server-side тестами регистрацию и базовый вызов.

Выполненные действия:

1. Для `Этапа 2` в `docs/mcp-tool-integration-spec.md` добавлены подэтапы внедрения, чтобы явно разделить подготовку feature-specific кода, регистрацию инструмента и локальную проверку server-side слоя.
2. В `build.gradle.kts` добавлена клиентская зависимость `ktor-client-content-negotiation` для HTTP-вызова внешнего API.
3. Рядом с сервером создан feature-specific пакет `mcp/server/fetchpost` со следующими частями:
   - `PostLookupClient`;
   - `JsonPlaceholderPost`;
   - `JsonPlaceholderPostLookupClient`;
   - helper-функции `fetchPostToolSchema()` и `fetchPostToolResult(...)`.
4. В качестве первого реального источника данных подключён публичный mock API `JSONPlaceholder`.
5. В `McpServer` зарегистрирован новый инструмент `fetch_post` с явным `inputSchema` и server-side обработчиком вызова.
6. Добавлены локальные server-side тесты:
   - на регистрацию `fetch_post` в сервере;
   - на успешный базовый вызов инструмента;
   - на ошибку валидации входного аргумента `postId`.
7. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- feature-specific код первого инструмента положить рядом с сервером отдельным пакетом, а не размазывать его по общим папкам;
- использовать `JSONPlaceholder` как минимальный внешний HTTP-источник без авторизации и дополнительной инфраструктуры;
- на серверном этапе вернуть читаемый текстовый результат через `CallToolResult`, а структурированную клиентскую модель вызова строить уже на следующих этапах.

Проверка:

- `createMcpServer(...)` теперь регистрирует `fetch_post` вместе с `ping` и `echo`;
- локальный вызов `fetchPostToolResult(...)` с корректным `postId` возвращает успешный `CallToolResult`;
- локальный вызов без `postId` возвращает ошибку валидации;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `05b3341` — детализация этапа 2 в ТЗ;
- `64f140e` — добавление инструмента `fetch_post` и клиента к mock API;
- текущий коммит этапа — добавление server-side тестов и фиксация результата этапа в implementation log.

Следующий шаг:

## Этап 3. Поддержка `tools/call` В MCP-Слое

Статус: завершён

Цель этапа:

- расширить проектный `mcp`-слой поддержкой вызова инструмента;
- ввести project-specific модели запроса и результата `tools/call`;
- локализовать в `mcp`-слое маппинг SDK-результата в проектный контракт.

Выполненные действия:

1. Для `Этапа 3` в `docs/mcp-tool-integration-spec.md` добавлены подэтапы внедрения.
2. В `mcp/model` добавлены новые project-specific модели:
   - `McpToolCallRequest`;
   - `McpToolCallResult`.
3. Интерфейс `McpClient` расширен новым сценарием `callTool(endpoint, request)`.
4. В `DefaultMcpClient` реализован вызов `tools/call` через официальный Kotlin MCP SDK.
5. Внутри `mcp`-слоя добавлен маппинг `CallToolResult` в проектную модель `McpToolCallResult`.
6. Локальные тестовые заглушки `McpClient` синхронизированы с новым контрактом.
7. Добавлены unit-тесты на:
   - сохранение контракта request/result моделей;
   - маппинг SDK-результата `CallToolResult` в проектную модель.
8. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- использовать в project-specific request обычный `Map<String, Any?>`, чтобы не протаскивать SDK-типы в верхние слои;
- на текущем этапе хранить результат вызова инструмента как список читаемых текстовых блоков, потому что server-side vertical slice уже возвращает текст через `CallToolResult`;
- оставить открытой возможность позже расширить проектную модель результата структурированными полями без изменения внешней границы `mcp`-слоя.

Проверка:

- `McpClient` теперь умеет не только `connect`, но и `callTool(...)`;
- `DefaultMcpClient` маппит `CallToolResult` в `McpToolCallResult`;
- `.\gradlew.bat test` завершается успешно.

## Этап 4. Интеграция С Агентом

Статус: завершён

Цель этапа:

- провести вызов MCP-инструмента через агентный слой;
- расширить agent-level контракты новым сценарием `tools/call`;
- подтвердить unit-тестами успешный и ошибочный agent-flow.

Выполненные действия:

1. Для `Этапа 4` в `docs/mcp-tool-integration-spec.md` добавлены подэтапы внедрения.
2. В `AgentRequest` добавлен новый сценарий `CallTool`.
3. В `AgentResponse` добавлен новый успешный ответ `ToolCallSuccess`.
4. В `DefaultAgent` реализована обработка agent-request на вызов инструмента через `McpClient.callTool(...)`.
5. В `DefaultAgentTest` добавлены unit-тесты:
   - на успешный вызов инструмента;
   - на ошибку при исключении в `McpClient.callTool(...)`.
6. В `ArchitectureContractsTest` добавлены проверки на сохранение agent-level контракта для нового request/response.
7. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- делать agent-level сценарий generic, а не жёстко привязанным к `fetch_post`, чтобы следующий этап `workflow` строился поверх универсального контракта;
- оставить общий `Failure` как единый способ сообщить об ошибке агента без преждевременного разрастания дерева agent-response;
- считать агент точкой, где проходит orchestration вызова инструмента, а не местом, где формируется CLI-представление результата.

Проверка:

- `DefaultAgent` теперь обрабатывает `AgentRequest.CallTool`;
- успешный agent-flow возвращает `AgentResponse.ToolCallSuccess`;
- ошибка в `McpClient.callTool(...)` преобразуется в `AgentResponse.Failure`;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `0040024` — детализация этапа 4 в ТЗ;
- текущий коммит этапа — интеграция вызова инструмента с агентным слоем, тестами и записью результата в implementation log.

## Этап 5. Интеграция С Workflow

Статус: завершён

Цель этапа:

- встроить вызов инструмента в `workflow`-слой;
- добавить отдельную workflow-команду и workflow-result для нового сценария;
- подтвердить тестами преобразование `workflow -> agent -> workflow result`.

Выполненные действия:

1. Для `Этапа 5` в `docs/mcp-tool-integration-spec.md` добавлены подэтапы внедрения.
2. В `workflow/command` добавлена новая команда `ToolPostCommand`.
3. В `workflow/result` добавлен новый результат `ToolCallResult`.
4. `DefaultWorkflowCommandHandler` расширен новым сценарием:
   - преобразует `ToolPostCommand` в `AgentRequest.CallTool`;
   - вызывает инструмент `fetch_post`;
   - преобразует `AgentResponse.ToolCallSuccess` в `ToolCallResult`;
   - преобразует `AgentResponse.Failure` в неуспешный `ToolCallResult`.
5. В `DefaultWorkflowCommandHandlerTest` добавлены unit-тесты:
   - на успешный workflow-сценарий вызова инструмента;
   - на ошибочный workflow-сценарий вызова инструмента.
6. В `ArchitectureContractsTest` добавлены проверки на сохранение новых workflow-контрактов.
7. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- на этапе `workflow` закрепить пользовательский вертикальный срез как `ToolPostCommand`, не пытаясь преждевременно строить общий DSL для всех инструментов;
- при этом agent-layer оставить generic, а workflow-layer сделать ответственным за привязку конкретной команды приложения к инструменту `fetch_post`;
- в неуспешном `ToolCallResult` хранить одно итоговое сообщение ошибки, а не пытаться на этом этапе строить отдельную иерархию ошибок.

Проверка:

- `DefaultWorkflowCommandHandler` теперь обрабатывает `ToolPostCommand`;
- успешный workflow-flow возвращает `ToolCallResult` с содержимым результата инструмента;
- агентная ошибка преобразуется в неуспешный `ToolCallResult`;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `36c51ca` — детализация этапа 5 в ТЗ;
- текущий коммит этапа — интеграция вызова инструмента в workflow-слой, тестами и записью результата в implementation log.
