# Техническое задание

## Название

Интеграция реального MCP-инструмента с агентом и приложением.

## Цель

Развить текущий sandbox-проект от сценария `connect -> initialize -> tools/list` до сценария, в котором:

1. на MCP server регистрируется прикладной инструмент;
2. инструмент имеет явное описание входных параметров;
3. инструмент возвращает структурированный или читаемый результат;
4. агент умеет вызывать этот инструмент через MCP;
5. приложение получает результат вызова и использует его в пользовательском сценарии.

Минимально успешный результат:

- агент делает `tools/call` к MCP server;
- MCP server выполняет инструмент;
- результат возвращается в агент;
- приложение выводит или иначе использует этот результат.

## Контекст

Проект уже умеет:

- поднимать локальный MCP server;
- публиковать инструменты `ping` и `echo`;
- подключаться к серверу через MCP client;
- получать `tools/list`;
- прогонять сценарий через архитектурную цепочку:
  - `presentation -> workflow -> agent -> mcp`;
- запускаться через direct launcher-артефакты и scripted e2e-проверки.

Следующий естественный шаг — не просто перечислять доступные инструменты, а реально вызывать один из них из приложения через агентный слой.

## Базовый Сценарий Для Этого ТЗ

Рекомендуемый стартовый вариант:

- взять `mock API` как внешний источник данных;
- зарегистрировать один новый прикладной tool поверх этого API;
- реализовать вызов tool из приложения;
- получить ответ и вывести его в CLI.

Почему `mock API` рекомендуется первым:

- меньше инфраструктурной сложности;
- проще изолировать сам принцип вызова MCP tool;
- легче тестировать и объяснять;
- не требует доступа к приватным сервисам или локальным утилитам.

Допустимые альтернативы:

- Git;
- CRM;
- Яндекс.Трекер;
- другой HTTP API.

Но для первого прохода приоритет у самого простого и воспроизводимого сценария.

## Функциональные Требования

- На сервере должен появиться минимум один новый прикладной MCP tool сверх `ping` и `echo`.
- Для этого tool должно быть явно задано:
  - имя;
  - описание;
  - входная схема `inputSchema`.
- Tool должен принимать минимум один входной параметр.
- Tool должен выполнять реальное действие:
  - запрос к внешнему API;
  - обращение к Git;
  - или другой допустимый источник данных.
- Tool должен возвращать результат через `CallToolResult`.
- В `mcp` слое клиента должна появиться поддержка вызова tool, а не только `tools/list`.
- Агент должен получить отдельный сценарий вызова инструмента.
- `workflow` слой должен получить отдельную команду для вызова инструмента.
- `presentation` слой должен уметь:
  - принять пользовательскую команду для вызова инструмента;
  - вывести результат в читаемом виде.
- Из приложения должен быть воспроизводимо доступен сценарий вызова инструмента.

## Нефункциональные Требования

- Нельзя ломать уже существующий сценарий `connect -> initialize -> tools/list`.
- Новый tool call не должен обходить агентный слой.
- `presentation` слой не должен работать с MCP SDK напрямую.
- `workflow` слой не должен знать деталей SDK и transport.
- Внешний API или mock API должен быть подключён так, чтобы его можно было при необходимости заменить или расширить.
- Для нового публичного orchestration-кода и важных интеграционных точек должна быть добавлена документация.

## Архитектурные Ограничения

### `presentation`

Должен:

- разобрать команду пользователя;
- создать `workflow`-команду;
- отформатировать результат.

Не должен:

- вызывать MCP SDK напрямую;
- знать детали `tools/call`.

### `workflow`

Должен:

- принимать отдельную команду вызова инструмента;
- делегировать выполнение агенту;
- преобразовывать агентный ответ в `workflow result`.

Не должен:

- знать детали HTTP transport;
- знать типы MCP SDK.

### `agent`

Должен:

- получить запрос на вызов инструмента;
- вызвать `McpClient`;
- вернуть структурированный агентный результат.

Именно агент должен быть точкой, где проходит вызов MCP tool.

### `mcp`

Должен:

- поддержать `tools/call`;
- инкапсулировать детали MCP SDK и маппинг результата.

## Ожидаемая Структура Изменений

На сервере:

- новый tool в `mcp.server`;
- helper-функции схемы и разбора аргументов;
- при необходимости отдельный клиент к выбранному API.

На стороне клиента:

- расширение `McpClient` и `DefaultMcpClient` поддержкой `callTool(...)`;
- новые модели запроса и результата tool call;
- новый агентный сценарий;
- новая `workflow`-команда и новый `workflow result`;
- обновление parser/formatter;
- scripted и e2e-проверка вызова инструмента.

## Рекомендуемый Вертикальный Срез

Если выбирать минимальный, но полноценный сценарий, рекомендуется такой поток:

1. пользователь вводит команду вроде `tool demo <arg>`;
2. CLI parser создаёт `workflow`-команду;
3. `workflow` handler передаёт её агенту;
4. агент вызывает `McpClient.callTool(...)`;
5. `McpClient` делает `tools/call` в MCP server;
6. MCP server обращается к mock API;
7. server возвращает результат;
8. агент получает и передаёт его вверх;
9. formatter выводит результат в CLI.

## Этапы Реализации

### Этап 1. Выбор Инструмента И Контракта

Цель:

- определить первый прикладной tool;
- зафиксировать входные параметры и формат результата;
- выбрать источник данных.

Результат этапа:

- понятен первый tool;
- понятна команда приложения;
- понятен ожидаемый результат.

Шаги этапа:

1. Выбрать источник данных:
   - предпочтительно `mock API`.
2. Определить первый прикладной tool.
3. Зафиксировать:
   - имя tool;
   - описание;
   - входные параметры;
   - структуру результата.
4. Зафиксировать пользовательскую команду приложения, которая будет запускать этот сценарий.

Подэтапы внедрения:

1. Детализировать `Этап 1` в ТЗ так, чтобы были явно видны:
   - порядок принятия решений;
   - минимальный ожидаемый результат каждого шага;
   - граница между выбором контракта и дальнейшей реализацией.
2. Отдельно зафиксировать выбор первого прикладного сценария:
   - источник данных;
   - имя инструмента;
   - пользовательскую ценность вызова.
3. Описать контракт первого инструмента в отдельном проектном документе:
   - входные параметры;
   - ожидаемую форму результата;
   - пользовательскую команду приложения.
4. Синхронизировать README и связанные документы так, чтобы следующий этап реализации опирался уже на зафиксированный контракт, а не на устные договорённости.

### Этап 2. Реализация Tool На Сервере

Цель:

- добавить на MCP server новый прикладной tool.

Результат этапа:

- tool зарегистрирован;
- `inputSchema` описана;
- tool возвращает результат.

Шаги этапа:

1. Добавить регистрацию нового инструмента в сервер.
2. Описать `ToolSchema`.
3. Реализовать разбор аргументов.
4. Подключить источник данных.
5. Вернуть результат через `CallToolResult`.
6. Добавить локальные тесты server-слоя на регистрацию и базовый вызов.

### Этап 3. Поддержка `tools/call` В MCP-Слое

Цель:

- расширить клиентский `mcp` слой поддержкой вызова инструмента.

Результат этапа:

- `McpClient` умеет вызывать инструмент;
- результат преобразуется в проектные модели.

Шаги этапа:

1. Расширить интерфейс `McpClient`.
2. Добавить реализацию `callTool(...)` в `DefaultMcpClient`.
3. Ввести проектные модели:
   - запрос вызова;
   - результат вызова;
   - ошибка вызова, если нужна отдельная модель.
4. Добавить unit-тесты на маппинг и формат контрактов `mcp` слоя.

### Этап 4. Интеграция С Агентом

Цель:

- провести вызов tool через агентный слой.

Результат этапа:

- агент умеет выполнять новый сценарий вызова инструмента.

Шаги этапа:

1. Добавить новый `AgentRequest`.
2. Добавить новый `AgentResponse`.
3. Реализовать обработку нового запроса в агенте.
4. Добавить unit-тесты на успешный и ошибочный сценарии.

### Этап 5. Интеграция С Workflow

Цель:

- встроить вызов инструмента в слой `workflow`.

Результат этапа:

- есть отдельная `workflow`-команда;
- есть `workflow result` для результата вызова инструмента.

Шаги этапа:

1. Добавить новую `workflow`-команду.
2. Добавить новый `workflow result`.
3. Расширить `WorkflowCommandHandler`.
4. Добавить unit-тесты на преобразование команд и результатов.

### Этап 6. Интеграция С Presentation

Цель:

- сделать вызов инструмента доступным из приложения.

Результат этапа:

- CLI понимает новую команду;
- formatter умеет выводить результат вызова инструмента.

Шаги этапа:

1. Расширить `CliCommandParser`.
2. Расширить `CliOutputFormatter`.
3. Обновить интерактивный help при необходимости.
4. Добавить unit-тесты parser/formatter.

### Этап 7. Сквозная Проверка И Документация

Цель:

- зафиксировать рабочий end-to-end сценарий вызова инструмента.

Результат этапа:

- scripted/e2e-проверка подтверждает вызов tool;
- README и журнал реализации обновлены.

Шаги этапа:

1. Обновить scripted smoke/e2e-сценарий.
2. Проверить:
   - `.\gradlew.bat test`
   - `powershell -ExecutionPolicy Bypass -File .\scripts\check-e2e.ps1`
   - ручной запуск через `собери проект`
3. Обновить README.
4. Завести или обновить отдельный журнал реализации под это ТЗ.

## Критерии Готовности

Реализация считается завершённой, если:

- на сервере зарегистрирован новый прикладной tool;
- у него есть описание входных параметров;
- tool реально возвращает результат;
- агент вызывает tool через MCP;
- приложение получает результат и отображает его;
- сценарий воспроизводим в ручном и scripted-режиме;
- существующий сценарий `connect -> initialize -> tools/list` остаётся рабочим.

## Что Не Входит В Объём

На текущем этапе в это ТЗ не входит:

- внедрение LLM;
- автоматический выбор tool моделью;
- многошаговый reasoning loop;
- полноценная поддержка нескольких внешних API сразу;
- сложная авторизация внешних корпоративных систем, если для первого вертикального среза можно взять более простой источник данных.
